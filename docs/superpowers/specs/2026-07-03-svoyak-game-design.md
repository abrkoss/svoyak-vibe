# «Своя игра» по локальной WiFi — дизайн

Дата: 2026-07-03

## Обзор

Веб-приложение для проведения игры типа «Своя игра» в локальной WiFi-сети.
Стек: **Node.js (Express) + Socket.IO** на бэкенде, **Vue 3 (Vite)** на фронтенде.
Один сервер отдаёт собранный фронт как статику и обслуживает WebSocket на одном
порту; все участники подключаются по `http://<LAN-IP>:PORT`.

Игра проходит без тестов (по решению заказчика) — упор на рабочий продукт.

## Роли и страницы

Приложение — единое Vue-приложение с тремя маршрутами:

1. **`/display`** — экран-показ (проектор/большой экран). Показывает табло и текущий
   вопрос **без правильного ответа**.
2. **`/host`** — интерфейс ведущего. Видит вопрос **с правильным ответом**, управляет
   игрой.
3. **`/player`** — интерфейс игрока (телефон/ноутбук). Кнопка ответа и (в финале) поля
   ставки и текстового ответа.

## Ключевые решения (из брейншторма)

- **Контент**: загружается из файла `data/questions.json`, который правится вручную
  перед игрой.
- **Кнопка ответа**: гонка с блокировкой — кто первый нажал, тот и отвечает; остальные
  блокируются. Ведущий видит победителя гонки. Игроки отвечают **устно**, ведущий
  подтверждает верно/неверно.
- **Счёт**: классический — у каждого вопроса есть цена; верно = +цена, неверно = −цена.
  Счёт ведёт сервер автоматически. Ведущий может вручную скорректировать очки кнопками
  +/−.
- **Финал**: после удаления тем ведущим остаётся 1 тема → игроки делают **скрытые
  ставки** → показывается вопрос → игроки вводят **текстовый ответ** → ведущий
  вскрывает ставки и ответы, подтверждает верно/неверно (±ставка).
- **Подключение игроков**: открывают URL сервера, вводят **имя** → попадают в лобби;
  ведущий видит список. Одна игра/комната (без кодов комнат).
- **Табло**: классическое (сетка тем × цен) видно на экране-показе; сыгранные вопросы
  гаснут. У ведущего — кликабельное табло для выбора вопроса.

```

├── server/

│   ├── socketHandlers.js # тонкий слой: события ↔ вызовы gameEngine
│   └── persistence.js    # загрузка questions.json, автосохранение состояния
├── data/
│   ├── questions.json    # контент (раунды/темы/вопросы)
│   └── questions.example.json

│   ├── src/
│   │   ├── pages/DisplayView.vue
│   │   ├── pages/HostView.vue
│   │   ├── pages/PlayerView.vue

│   │   └── router.js
```

Принцип: вся логика игры сосредоточена в `gameEngine.js` (чистые функции над
состоянием, без сети). Socket-слой только транслирует события. Сервер слушает
`0.0.0.0` и при старте печатает `http://<LAN-IP>:PORT`.

## Модель данных контента (questions.json)

```json
{
  "title": "Моя своя игра",
  "rounds": [
    {
      "name": "Раунд 1",
      "type": "normal",
      "themes": [
        {
          "name": "История",
          "questions": [
            { "value": 100, "text": "Вопрос...", "answer": "Ответ..." },
            { "value": 200, "text": "...", "answer": "..." }
          ]
        }
      ]
    },
    {
      "name": "Финал",
      "type": "final",
      "themes": [
        { "name": "Космос", "text": "Финальный вопрос...", "answer": "..." }
      ]
    }
  ]
}
```

- Раундов может быть несколько (`type: "normal"`).
- Последний раунд — финал (`type: "final"`): каждая тема содержит один вопрос.

## Состояние игры (в памяти + автосохранение)

- `players`: `[{ id, name, score, connected }]`
- `phase`: `lobby | board | question | buzzed | answered | reveal |
  final_remove | final_bets | final_answers | final_reveal | finished`
- `currentRoundIndex`
- `answeredQuestions`: множество сыгранных клеток табло текущего раунда
- `currentQuestion`: `{ roundIndex, themeIndex, questionIndex, buzzerOpen, buzzedPlayerId }`
- Финал: `removedThemes[]`, `bets{playerId: n}`, `answers{playerId: text}`

Состояние автосохраняется в `data/game-state.json`, чтобы перезапуск сервера не терял
счёт. `playerId` хранится в `localStorage` игрока → переподключение восстанавливает
игрока.

## Поток одного вопроса (обычный раунд)

1. Ведущий кликает клетку табло → вопрос появляется на показе и у ведущего (у него — с
   ответом). `phase = question`.
2. Ведущий «Открыть кнопки» → `buzzerOpen = true`.
3. Первое нажатие `buzz` блокирует остальных; сервер атомарно фиксирует победителя
   гонки; показ и ведущий видят имя. `phase = buzzed`.
4. Игрок отвечает устно → ведущий жмёт:
   - «Верно» → +цена победителю, вопрос гаснет, возврат к табло.
   - «Неверно» → −цена, кнопки открываются заново для остальных (`buzzerOpen = true`,
     ответивший исключается из этой гонки).
5. Когда все вопросы раунда сыграны — ведущий жмёт «Следующий раунд».

## Поток финала

1. Переход в финальный раунд → на показе список финальных тем. `phase = final_remove`.
2. Ведущий удаляет темы кликом (по решению игроков устно) до одной оставшейся.
3. `phase = final_bets`: игроки вводят скрытую ставку. Ставки скрыты на показе.
4. Ведущий раскрывает вопрос → `phase = final_answers`: игроки вводят текстовый ответ.
5. `phase = final_reveal`: ведущий вскрывает ставки и ответы по игрокам, подтверждает
   верно/неверно → ±ставка. По завершении — `phase = finished`, итоговый счёт.

## Обработка ошибок и надёжность

- Переподключение игроков — Socket.IO + `playerId` в localStorage.
- Отвалившийся игрок помечается `connected: false`, счёт сохраняется, восстанавливается
  при возврате.
- Гонка кнопок решается атомарно на сервере: первый `buzz` при `buzzerOpen === true` —
  победитель, остальные игнорируются.
- Отрицательный счёт допускается (как в оригинале). Ставки в финале не превышают
  разумных пределов; валидация на сервере.

## Запуск

- `npm run dev` — разработка (Vite dev-сервер + Node сервер).
- `npm run build` + `npm start` — сборка фронта и запуск боевого сервера, печатает
  LAN-URL для игроков.
